**Part A — FavouriteSwipeScreen tap handler:**

`FavouriteCard` does not contain `legalities`, `prices`, `oracleText`, or other fields
required by `CardDetailScreen`. Constructing a partial `MagicCard` would show blank/N/A
for all metadata — a poor experience. Instead, **use getCardById** (added in plan 04-01)
to fetch a full `MagicCard` before navigating. This is consistent with D-01's intent
(the detail screen itself does not re-fetch; the tap handler in the swipe screen fetches once).

**Resolution of RESEARCH.md Pitfall 6 (open question):** Use Option D from RESEARCH.md —
the tap handler in FavouriteSwipeScreen calls `getCardById` on the repository, then
navigates with the full `MagicCard`. D-01 ("no re-fetch") means the detail screen does not
re-fetch; it does not prohibit the tap handler from fetching. The CLAUDE.md rule
"No direct Scryfall calls from presentation layer" is honoured — fetch goes through the
repository/provider, not `ScryfallApiClient` directly.

**Implementation pattern:**

```dart
// In the widget where _FavouriteCardFace / card face is displayed,
// wrap with GestureDetector:

GestureDetector(
  onTap: () async {
    // Fetch full MagicCard to populate all detail screen fields (Pitfall 6 resolution).
    // Goes through repository — no direct API call from presentation layer.
    final result = await ref.read(cardRepositoryProvider).getCardById(favouriteCard.id);
    result.fold(
      onSuccess: (card) {
        if (context.mounted) {
          context.go('/card/${card.id}', extra: card);
        }
      },
      onFailure: (failure) {
        // Show snackbar on failure — user stays in swipe view.
        if (context.mounted) {
          ScaffoldMessenger.of(context).showSnackBar(
            const SnackBar(content: Text('Could not load card details. Try again.')),
          );
        }
      },
    );
  },
  child: _FavouriteCardFace(card: favouriteCard), // preserve existing args
)
```
